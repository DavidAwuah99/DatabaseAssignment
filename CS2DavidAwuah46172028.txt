INSERT can be maintained without accessing Part(base table) because the partCost > 1000 filter only needs the new rows data and the second check DISTINCT only needs the view's own stored records/rows - never the base table Part itself.

DELETE requires accessing the base table Part. A row (partNo, contract, partCost) is deleted. PartNo might have had other rows that are priced above 1000 pounds in the view. The only way to check remaining parts that matches the partNo is by querying the base table Part.

UPDATE behaves like an INSERT or DELETE based on the bound of the ExpensivePart view. If you're adding to the view it's like an INSERT but if you're removing from the view it acts like a DELETE. Here it really depends on which operation is being done and that determines if the base table is referenced or not
